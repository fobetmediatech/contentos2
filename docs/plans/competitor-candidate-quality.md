## Problem statement

The competitor pipeline returns a final 10 (top 5 + trending 5) that has three recurring
failure modes in real runs:

- **Precision**: off-subniche or loosely related accounts slip into the 10.
- **Recall**: strong, obvious accounts never enter the pool (closed relatedProfiles graph,
  sparse hashtags).
- **Selection/sorting**: the 10 are relevant but mis-tiered, or not the best 10 from a good pool.

Identity verification (impostors/homonyms) is NOT a top pain. The scrape-verify +
`matchesIntendedIdentity` gate already handles that. The pain is the relevance and ranking
of real accounts.

The user wants candidates "gathered and verified by web search," sorted by niche AND
subniche, with only the relevant 10 delivered.

## Where the ceiling actually is

The pool has six sources, but **selection is one flat Gemini call judging niche-fit from a
~120-character bio snippet**, then tiering by follower count and ER. Thin signal in,
mediocre 10 out. That single fact explains all three pains:

- Precision: a 120-char bio cannot separate "entrepreneurship coach" from a trading/crypto
  hustle bio that name-drops the same words.
- Recall: the pool is capped by the graph walk plus one seed call, so closed-graph niches
  stay thin.
- Sorting: top-vs-trending is a follower heuristic, not grounded in who the niche's real
  leaders are.

## Premises (agreed in session)

1. The bottleneck is per-candidate signal, not raw pool size. More signal for the selector,
   plus splitting "is it relevant" from "which tier," moves precision and sorting more than
   scraping more accounts.
2. Recall and precision only rise together when new candidates are **subniche-targeted**.
   Generic web search adds junk. The new niche briefing already names subniches and leaders,
   so fanning web-seeding out per subniche adds recall that is relevant by construction.
3. Spend "verified by web search" on the **margin**, not every candidate. Confirming obvious
   leaders wastes calls. Confirming borderline accounts is where a web lookup flips the
   decision.

## Baseline (what exists today, after the current branch work)

- `generateNicheSeeds(...)` -> `{ briefing, candidates }` via ONE web-grounded call
  (`buildNicheSeedPrompt` returns `{ niche_brief, accounts }`).
- `discoverSpeculativeSources` -> `{ profiles, nicheBriefing }` (knowledge seed leg + IG
  keyword-search leg).
- `discoverCompetitors` -> merges graph walk + speculative, dead-account gate ->
  `{ inputProfiles, candidateProfiles, nicheBriefing }`.
- `analyzeCompetitors` -> `callGeminiWithSchema(COMPETITOR_SCHEMA)` ranks the flat pool into
  top5/trending5, hallucination filter (must be in pool), relaxed underfill top-up. The
  `nicheBriefing` is injected as a WEB RESEARCH block.

## Recommended approach: B (= A + margin web verification)

Three layers. Each degrades gracefully to the layer below on failure.

### Layer 1 — Subniche-targeted recall (the A recall piece)

1. Extend the grounded seed response to also return `subniches: string[]` (the model already
   reasons about them; capture them structured next to `niche_brief` and `accounts`). Parse
   in `parseNicheSeedResult`.
2. Capture the briefing's named leaders as direct candidate handles (they are already named
   in prose; have the prompt list them in `accounts` so they enter the scrape-verify path).
3. **Budget-gated fan-out**: if the merged pool is thin (fewer than the target after the
   normal sources) OR the niche is broad, fire ONE additional grounded seed call per subniche
   (cap at 2-3 subniches), each scoped to that subniche, then union + dedup against
   `seenInput` and already-scraped handles. Each batch goes through the same scrape-verify +
   identity gate as today.
   - Hard cap total grounded seed calls per run at 3 (CR-1 latency budget).

### Layer 2 — Two-stage selection (the A precision + sorting piece)

Replace the single ranking call with two stages:

- **Stage 1, relevance gate** (precision): a structured call that scores each pool candidate
  on subniche fit using the briefing + its EXCLUDE list. Output per candidate: `fit` (0=off,
  1=borderline, 2=clear) + the matched subniche. Drop `fit==0`. This is a cheap classification
  pass — default model, `thinkingBudget: 0`. It uses `responseSchema` (no `googleSearch`, so
  the JSON-mode + grounding conflict does not apply here).
- **Stage 2, tier + rank survivors** (sorting): rank survivors into top5/trending5 with a
  leader-vs-riser definition anchored on the briefing's named leaders. "Top" = established
  leaders (named in the briefing, or the existing 500k+ ESTABLISHED label). "Trending" =
  genuine risers (strong ER, mid-size, recent activity). Reuses most of the current
  `buildCompetitorPrompt` logic, minus the relevance work now done in Stage 1.

### Layer 3 — Margin web-verification dossiers (the B distinctive piece)

- After Stage 1, collect the **borderline** set: `fit==1`, plus knowledge/search-sourced
  accounts whose bio does not clearly confirm the niche (the existing SOURCE PRIORITY
  "must CONFIRM" cases). Cap at ~6 handles.
- For that set only, ONE batched web-grounded call (`callGeminiGroundedJson`) returns a
  one-line "known for" + verdict (`fits | off-niche | unsure`) per handle. One web search for
  the whole batch, not one per handle.
- Use the existing `groundingMetadata` activation check: if grounding did not actually run,
  skip dossiers and fall back to A.
- Feed dossiers into Stage 2 as extra per-candidate signal, and drop any account the dossier
  verdicts `off-niche` even if its bio passed Stage 1.

### Layer 4 — Exactly-10 guarantee + transparency

- After Stage 2, fill to 10 (5+5) where the pool allows. The relaxed underfill top-up still
  applies but is now relevance-gated, so it cannot add filler.
- Cross-check the briefing's named leaders against the final 10. If a named leader is missing,
  `devLog` why (not scraped / filtered / off-pool). This surfaces recall gaps for debugging.
  DEV-only, never user-facing (C3).

## Cost and latency controls (load-bearing for B)

Worst-case grounded web searches per run:
- 1 existing seed call
- 0-2 subniche fan-out calls (only when the pool is thin), capped so total seed calls <= 3
- 0-1 dossier call (only when the borderline set is non-empty), batched, <= 6 handles

Net new structured (non-grounded) call: 1 relevance gate (cheap, `thinkingBudget: 0`).

Everything is gated and degrades: grounding fails -> fall back to A; A fails -> today's
behavior. No path makes a run slower or more expensive when the cheap path already produced
a clean 10.

## Approaches considered

### Approach A: Subniche fan-out + two-stage select
Layers 1 + 2 only. Effort: human ~1-2 days / CC ~30-45 min. Hits all three pains with mostly
prompt + plumbing on existing code. Leaves some borderline filler because there is no
per-candidate web verification.

### Approach B: Web-grounded candidate dossiers (RECOMMENDED)
A + Layer 3 margin verification. Effort: human ~3-4 days / CC ~1-2 hrs. Delivers the literal
"gathered and verified by web search" requirement, bounded to the margin so cost stays sane.
Highest output precision.

### Approach C: Confirmed-seed snowball + judge panel
Recall by snowballing relatedProfiles from confirmed-good accounts; selection by a 2-3 pass
judge panel reconciled to a consensus 10. Effort: human ~3-5 days / CC ~1-2 hrs. Most
variance-proof selection, but the most calls/latency and the biggest change to the ranking
core. Snowball can drift into adjacent niches if the seed set is slightly off. Deferred —
the snowball recall idea is a good future add-on to B if recall is still thin after Layer 1.

## Recommended approach

B, built in two shippable steps: ship A (Layers 1+2) first behind the existing flow, measure
against the eval set, then add Layer 3 dossiers. This front-loads the cheap 80% and adds the
web-verification layer once the two-stage selection is proven.

## Open questions

- Stage 1 model: default `gemini-2.5-flash` (cheap) vs premium. Start with default + thinking
  off; promote to premium only if precision is still short on the eval set.
- Borderline threshold and dossier cap (start `fit==1`, cap 6) — tune against the eval set.
- Subniche fan-out trigger: thin-pool only, or always for broad niches? Start thin-pool-only
  to protect latency.
- Should an `off-niche` dossier verdict feed back into the corpus as a soft negative signal
  for future runs? (Possible Phase 3 self-training tie-in. Out of scope for v1.)

## Success criteria

- Precision: near-zero off-subniche filler in the final 10 across a 10-niche eval set.
- Recall: briefing-named leaders that exist on Instagram appear in the pool >= 90% of runs.
- Sorting: on manual check, top5 are established names and trending5 are genuine risers.
- Cost: <= 4 grounded calls/run; p50 added latency under ~15s vs today.
- No regression on the existing unit suite.

## Distribution

Internal web feature. Existing Vercel deploy pipeline covers it. No new distribution channel.

## Dependencies

- Builds directly on the current branch (`generateNicheSeeds -> { briefing, candidates }`,
  the WEB RESEARCH prompt block, the grounding activation check). Land that first.
- No schema/DB migration. The competitor `analysisStore` is transient.

## The assignment (do this before building)

Assemble a 10-niche evaluation set with, for each niche: the hand-labeled "ideal 10" (which 5
you would call top, which 5 trending) AND 3-5 known-good accounts the tool currently MISSES.
That labeled set is the only thing that tells you whether B actually beats A, and it becomes
the regression fixture for the two-stage selection. Without it you are tuning blind.

## What I noticed

- You picked Precision, Recall, and Sorting but explicitly NOT Verification. That is a sharp
  read of your own system: it says your identity gate works and the real gap is relevance and
  ranking of real accounts, not catching impostors. Most people would have lumped all four
  together.
- Your framing "verified / gathered by the web search and then only the relevant 10" already
  encodes the right architecture: gather wide (recall), verify on the way in (precision),
  deliver tight (selection). The design just makes that explicit and bounds the cost.
